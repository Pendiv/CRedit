────────────────────────────────────────────────────────────────
  1. 背景と動機
───────────────────────────────────────────────────────────────

KubeJS でレシピを書くのは面倒。特に GTCEu のように独自レシピ
タイプを大量に持つ環境では、メソッド名(.itemInputs, .EUt,
.duration 等)やスロット構成を都度調べる必要があり、開発体験が
悪い。

この面倒を UI で解消する。コマンド一発でレシピエディタを立ち
上げ、直感的な操作でレシピを組み立て、KubeJS コードとして
ダンプする。


────────────────────────────────────────────────────────────────
  2. 基底コンセプト
────────────────────────────────────────────────────────────────

  「JEI の上に乗った、逆向きのレシピエディタ」

JEI が「レシピを見る」ツールなら、これは「JEI の視点で、JEI の
語彙で、レシピを書く」ツール。

比喩すると: 親(JEI)を完全に信頼しきっているからこそ反抗できる、
反抗期の子。JEI に全面依存することで、JEI が想定しない逆向きの
使い方が可能になる。

具体的には:
  - レシピタイプ検出を自前でやらない → JEI のカテゴリ一覧を使う
  - UI レイアウトを自前で作り込まない → JEI の IRecipeCategory から
    レイアウト情報を借用する
  - アイテム/液体/タグの検索を自前で実装しない → JEI の検索機能を
    そのまま呼び出す

この「全面依存」により、MOD が対応できる範囲は「JEI に登録されて
いる全てのレシピカテゴリ」になる。GTCEu の全機械も、バニラの
クラフトもスメルティングも、他 MOD のレシピも、区別なく扱える。

  3. スコープ
────────────────────────────────────────────────────────────────

【やること (v1)】
  - JEI に登録されたレシピカテゴリの列挙・選択 UI

  - GTCEu 全レシピタイプ(JEI カテゴリ経由で自動対応される想定)

────────────────────────────────────────────────────────────────
  5. 起動タイミングと初期化
────────────────────────────────────────────────────────────────

5.1 設計の肝: コマンド起点+遅延実行

  ユーザー操作を起点にすることで、MOD 初期化順序の呪いを
  原理的に回避する。

  - MOD 読み込み時は @JeiPlugin とコマンド登録のみ
  - JEI ランタイム確立時に IJeiRuntime を static 保持
  - それ以外の処理は /craftpattern 実行時まで遅延

  結果: コマンド実行時点では JEI もレシピも全て準備済み、という
  前提で動ける。事前準備がほぼゼロ、初期化競合なし。

5.2 コマンド登録形式

  クライアント側コマンド (RegisterClientCommandsEvent)

  理由:
   - このMODは純粋にクライアント機能
   - サーバー⇔クライアントのパケット通信不要
   - Screen 表示が直接できる
   - シングルプレイ/マルチ問わず統一的に動く

5.3 IJeiRuntime のライフサイクル

  - volatile フィールドで static 保持
  - onRuntimeAvailable が呼ばれるたびに更新
  - ワールド退出・再入場でも JEI が再通知してくれる
  - 未初期化時(jeiRuntime == null)はエラー応答

5.4 エラー応答(未初期化時)

  タイトル画面やJEI未初期化状態で /craftpattern が叩かれたら、
  エラーメッセージを返して終了:

    [CraftPattern] JEI is not ready yet. Enter a world first.

  6. コマンド仕様
────────────────────────────────────────────────────────────────

6.1 形式

  /craftpattern                     -- ピッカー画面を開く
  /craftpattern <categoryId>        -- 指定カテゴリの Screen に直行
  /craftpattern <categoryId> [opts] -- オプション指定付き直行

6.2 引数補完

  第1引数: JEI に登録された全カテゴリ ID をサジェスト
           入力中の文字列で前方一致 + namespace フィルタ

6.3 権限

  クライアント側コマンドなので権限チェック不要
  (実体としてはクライアントのファイル操作のみ)

7.1 Picker Screen (カテゴリ選択画面)

  形式: リストレイアウト、上部に検索欄

  ┌─────────────────────────────────────┐
  │ Select Recipe Category          [×] │
  │ ┌─────────────────────────────────┐ │
  │ │ [search...                    ] │ │
  │ └─────────────────────────────────┘ │
  │ ┌─────────────────────────────────┐ │
  │ │ > minecraft:crafting_shaped     │ │
  │ │   minecraft:crafting_shapeless  │ │
  │ │   minecraft:smelting            │ │

  │ │   ...                           │ │
  │ └─────────────────────────────────┘ │
  └─────────────────────────────────────┘
                                   ┌────┐
                                   │JEI │
                                   └────┘

  操作: クリック or Enter で該当カテゴリの RecipeBuilderScreen へ

7.2 RecipeBuilderScreen (各カテゴリ専用)

  JEI の IRecipeCategory からレイアウトを借用して生成。
  特殊対応が必要な場合のみ自前実装で上書き。

  共通要素:
   - Recipe ID 入力欄 (未入力なら自動生成)
   - タグモード切替ボタン(各スロットごと)
   - Preview JS ボタン
   - Dump to file ボタン
   - Cancel ボタン
   - 出力先パス表示

  カテゴリ別レイアウト(v1):
   - crafting_shaped:    3×3入力 + 出力1
   - crafting_shapeless: 最大9入力 + 出力1
   - smelting:           入力1 + 出力1 + xp + cookingTime

7.3 JEI 連携機能

  スロットへのアイテム配置方法(全部併用):
   1. JEI からの D&D (Ghost Ingredient Drag API)
   2. JEI 項目を Shift+右クリ でスロットに送る
   3. スロットをクリック → JEI 検索にフォーカス
   4. インベントリからの D&D

  利用する JEI API:
   - IRecipeCategory          レイアウト取得
   - IIngredientManager       アイテム/液体/タグ統合アクセス
   - IRecipeManager           レシピ情報
   - IJeiRuntime              ランタイム全般
   - IGhostIngredientHandler  D&D 受け取り
   - IIngredientFilter        検索欄連携
   - お気に入り API           お気に入りからの配置


────────────────────────────────────────────────────────────────
  8. データモデル
────────────────────────────────────────────────────────────────

8.1 RecipeDraft (中央契約点)

  sealed interface で型安全に。カテゴリごとに専用 record。

    RecipeDraft (sealed)
     ├ ShapedDraft
     │   pattern: String[]           (例: ["III", " S ", " S "])
     │   key: Map<Char, IngredientSpec>
     │   result: IngredientSpec
     ├ ShapelessDraft
     │   ingredients: List<IngredientSpec>
     │   result: IngredientSpec
     └ SmeltingDraft
         input: IngredientSpec
         result: IngredientSpec
         xp: float
         cookingTime: int

  全Draftが共通で持つ:
   - categoryId: ResourceLocation
   - recipeId: ResourceLocation

8.2 IngredientSpec (スロット1個の中身)

  sealed interface。 Item/Tag/Fluid の3種。

    IngredientSpec (sealed)
     ├ ItemSpec    (item: ResLoc, count: int, nbt: CompoundTag?)
     ├ TagSpec     (tag: ResLoc, count: int)
     └ FluidSpec   (fluid: ResLoc, amount: int)

8.3 sealed にした理由

  カテゴリ追加時、全 Emitter が対応しているかコンパイラが検査
  してくれる。パターンマッチで switch 漏れを検出可能。


────────────────────────────────────────────────────────────────
  9. Dump 仕様 (KubeJS 書式出力)
────────────────────────────────────────────────────────────────

9.1 KubeJS 前提の明示

  MOD 自体は KubeJS に非依存(mods.toml で mandatory=false)。
  ただし生成される .js は KubeJS 書式。

  明示場所:
   1. 生成ファイル先頭ヘッダコメント (必ず目に入る)
   2. README
   3. Dump 時のチャット通知

9.2 Recipe ID 命名規約

  - ユーザー入力優先
  - 未入力時: morecommands:generated/<result_item_path>
    例: minecraft:iron_pickaxe → morecommands:generated/iron_pickaxe
  - 重複時: _2, _3 と連番

9.3 ファイル配置規約

  パス: kubejs/server_scripts/generated/<category_path>.js

  カテゴリごとに1ファイル、追記形式。

  namespace 別扱い:
   - minecraft → namespace 省略
       generated/crafting_shaped.js
   - minecraft 以外 → namespace サブフォルダ
       generated/gtceu/electric_blast_furnace.js
       generated/mekanism/crushing.js

9.4 ファイル骨格 (新規作成時)

    ┌─────────────────────────────────────────────────────┐
    │ // ===============================================  │
    │ // Generated by MoreCommands /craftpattern          │
    │ // FORMAT: KubeJS (https://kubejs.com/)             │
    │ // Category: minecraft:crafting_shaped              │
    │ // Place under kubejs/server_scripts/ to work.      │
    │ // Run /reload or /kubejs reload server_scripts.    │
    │ // ===============================================  │
    │                                                     │
    │ ServerEvents.recipes(event => {                     │
    │                                                     │
    │     // @morecommands:insert-marker (do not remove)  │
    │ });                                                 │
    └─────────────────────────────────────────────────────┘

  ヘッダは最小限: 生成日時 + カテゴリ id + KubeJS 注記のみ。

9.5 追記ロジック

    ファイル存在する?
     │
     NO → 新規作成(骨格をそのまま作って空レシピ追記モードへ)
     │
     YES → マーカーコメント探す
            │
            見つかった → マーカー直前に挿入
            │
            見つからない → 末尾の `});` を探す
                         │
                         見つかった → その直前に挿入
                         │
                         見つからない → フォールバック
                                       → 連番新ファイル作成
                                       (例: crafting_shaped_2.js)
                                       → チャットで通知

  マーカー + フォールバック の二段構え。マーカーがユーザーの
  手動編集で消えていても、`});` 検索で大抵救える。
  両方失敗した場合のみ連番新ファイルに逃げる。

9.6 手動編集時のロバスト性

  原則: 「MoreCommands が生成したファイルでも、ユーザーの手動
        編集は尊重する。無理そうなら素直に諦めて別ファイル」

  これにより以下が保証される:
   - 既存コードを絶対に壊さない
   - ユーザーが安心して生成後に編集できる
   - 最悪でも連番ファイル作成で Dump 自体は成功する

9.7 生成コード例

  ■ shaped
    event.shaped(
        Item.of('minecraft:iron_pickaxe'),
        ['III', ' S ', ' S '],
        {I: 'minecraft:iron_ingot', S: 'minecraft:stick'}
    ).id('morecommands:generated/iron_pickaxe')

  ■ shapeless
    event.shapeless(
        Item.of('minecraft:bread'),
        ['minecraft:wheat', 'minecraft:wheat', 'minecraft:wheat']
    ).id('morecommands:generated/bread_custom')

  ■ smelting
    event.smelting(
        Item.of('minecraft:iron_ingot'),
        'minecraft:raw_iron'
    )
        .xp(0.7)
        .cookingTime(200)
        .id('morecommands:generated/iron_smelting')

9.8 チャット通知

  通常時:
    [CraftPattern] Dumped to: generated/crafting_shaped.js
    [CraftPattern] Run /reload or /kubejs reload server_scripts to apply.

  フォールバック時:
    [CraftPattern] ⚠ Could not append to: generated/crafting_shaped.js
    [CraftPattern] ⚠ Reason: Marker missing, fallback insertion point not found
    [CraftPattern] → Created new file instead: generated/crafting_shaped_2.js
    [CraftPattern] Run /reload to apply.


────────────────────────────────────────────────────────────────
  10. リロード方針
────────────────────────────────────────────────────────────────

Dump 後のリロードはユーザー任せ。自動実行はしない。

理由:
 1. ユーザーの意図を尊重(連続Dumpで最後にまとめてリロード等)
 2. 失敗しない(自動リロードで壊れたらMODの責任になる)
 3. 責任範囲の明確化

実装: チャットでリロードコマンドを案内するのみ。


────────────────────────────────────────────────────────────────
  11. Emitter アーキテクチャ
────────────────────────────────────────────────────────────────

11.1 基本方針

  カテゴリごとに Emitter 実装を1つずつ用意。
  EmitterRegistry が categoryId → Emitter の解決を担う。

  カテゴリ追加 = Emitter 追加、の閉じた修正で済む構造。

11.2 共通化レベル

  v1 は「直接文字列生成」で行く。JsonElement 等の中間表現は
  使わない。理由: KubeJS 書式は vanilla JSON の 1:1 変換では
  ないため、中間を挟むメリットが小さい。YAGNI。

  共通化するのは以下のみ:
   - 共通ヘッダ(日時・カテゴリid・KubeJS注記)
   - ServerEvents.recipes ラッパ
   - .id(...) 句
   - IngredientEmitter (Item/Tag/Fluid → JS 文字列)

11.3 未対応カテゴリの扱い

  EmitterRegistry.get() が空を返した場合、
  UnsupportedCategoryException で明示的にエラー。

  Screen 側では Dump ボタン押下時にキャッチして通知:
    [CraftPattern] This category is not yet supported for dumping.
    [CraftPattern] Category: gtceu:some_new_type
